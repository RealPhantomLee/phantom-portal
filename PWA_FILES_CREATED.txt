INTEGRATION REQUIREMENTS
========================

For Push Notifications to work:

1. Generate VAPID keys (one-time):
   pip install web-push
   python3 << 'EOF'
   from web_push import generate_vapid_keys
   keys = generate_vapid_keys()
   print("Public:", keys['public'])
   print("Private:", keys['private'])
   EOF

2. Update phantom.config.yaml:
   web_push:
     vapid_private_key: "GENERATED_PRIVATE_KEY"
     vapid_public_key: "GENERATED_PUBLIC_KEY"
     vapid_email: "admin@example.com"

3. Install backend dependency:
   pip install web-push

4. Build frontend:
   cd frontend && npm run build

5. Motion events can now send push notifications:
   - MQTT motion events are received
   - They trigger WebSocket broadcasts to connected clients
   - They can also trigger push notifications to subscribed users
   - Push router has helper: send_motion_notification()

USAGE IN REACT COMPONENTS
==========================

Example with usePWA hook:

```typescript
import { usePWA } from './pwa/usePWA';

export function SettingsPanel() {
  const [state, actions] = usePWA();

  return (
    <div>
      {/* Install button */}
      {state.canInstall && (
        <button onClick={() => actions.showInstallPrompt()}>
          Install App
        </button>
      )}

      {/* Notifications toggle */}
      {state.notificationsSupported && (
        <button
          onClick={() =>
            state.isSubscribed
              ? actions.unsubscribeFromPushNotifications()
              : actions.subscribeToPushNotifications()
          }
          disabled={state.isSubscribing || state.isUnsubscribing}
        >
          {state.isSubscribed ? 'Disable' : 'Enable'} Notifications
        </button>
      )}

      {/* Test notification */}
      {state.isSubscribed && (
        <button onClick={() => actions.testPushNotification()}>
          Test Notification
        </button>
      )}
    </div>
  );
}
```

CACHING STRATEGY
================

CACHE-FIRST (Assets):
  Files: .js, .css, .html, .png, .svg, .ico, .woff, .woff2
  Behavior:
  1. Check browser cache
  2. If found, serve from cache
  3. If not found, fetch from network
  4. Cache successful responses
  5. On network failure, serve from cache if available

NETWORK-FIRST (API):
  Paths: /api/*
  Behavior:
  1. Try to fetch from network
  2. If successful, cache the response
  3. On network failure, try cache
  4. If no cache, return offline error

NO CACHE (WebSocket):
  Protocols: ws://, wss://
  Always real-time, never cached

SERVICE WORKER LIFECYCLE
========================

INSTALL:
- Pre-cache essential files (index.html)
- Skip waiting to activate immediately
- Clean up old caches

ACTIVATE:
- Delete outdated cache versions
- Claim all clients

FETCH:
- Route to appropriate cache strategy
- Handle API errors gracefully
- Support offline mode

PUSH:
- Display rich notifications
- Include camera thumbnails
- Include AI narration text
- Navigate to relevant page on click

SYNC (Background):
- Triggered when back online
- Sync pending note updates
- Retry failed requests

TROUBLESHOOTING
===============

Service Worker not registering:
- Check HTTPS or localhost
- Verify MIME type is application/javascript
- Look for errors in Console
- Check DevTools > Application > Service Workers

Push notifications not showing:
- Verify Notification.permission === 'granted'
- Check VAPID keys in config
- Test endpoint: await push.testPushNotification()
- Check server logs for send failures
- Verify notification.json payload format

Caching not working:
- Clear cache: DevTools > Application > Clear site data
- Check cache strategy matches path patterns
- Update cache version to invalidate (CACHE_ASSETS_v1 → v2)
- Verify fetch event is being intercepted

App not installing:
- Check manifest.json is valid
- Verify HTTPS or localhost
- Check icons are accessible at URLs
- On iOS, use "Add to Home Screen" instead of prompt
